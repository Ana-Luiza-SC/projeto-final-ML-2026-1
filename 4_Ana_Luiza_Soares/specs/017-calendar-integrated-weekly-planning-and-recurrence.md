# Spec 017 - Calendar-Integrated Weekly Planning and Recurrence

> Canonical language: English
> Translation: [../spec_traduzido/017-calendar-integrated-weekly-planning-and-recurrence.md](../spec_traduzido/017-calendar-integrated-weekly-planning-and-recurrence.md)
> Status: implemented
> Last reviewed: 2026-07-13

> **Relationship notice:** Spec 018 refines the frontend information architecture and planning explanations while preserving this specification's planned-block, recurrence, confirmation, and calendar contracts.

## Problem Statement

The current weekly planning and calendar experience mixes three different concepts:

- manual priority selection;
- generated study sessions that may not be persisted as calendar events;
- a month calendar plus a separate weekly agenda that duplicates information.

This makes the product harder to audit and weakens the distinction created by Specs 014, 015 and 016:

- priority determines what should be studied first;
- planned calendar blocks reserve time;
- study activities record the learner's actual execution and feedback.

Spec 017 replaces legacy weekly study session generation with calendar-integrated planned study blocks and adds recurrence to manual academic events.

## Scope

This spec defines:

- daily availability input by weekday and time window;
- backend-derived weekly availability totals;
- deterministic priority ranking owned by the backend;
- preview and confirmation for planned study blocks;
- persistence of confirmed planned blocks as academic calendar events;
- Month and Week calendar views without a duplicated weekly agenda;
- recurrence rules for manual academic events;
- API, persistence, migration, frontend behavior, guardrails and tests.

## Out of Scope

- Google Calendar, Outlook or external calendar sync.
- Login to SIGAA or authenticated scraping.
- LLM-defined priority scores, dates, weights, grades, mastery or content.
- Automatic creation of actual study activities.
- Pomodoro-based calendar generation.
- Recurring study-plan blocks beyond explicitly recurring availability used for one generated preview.
- Infinite materialization of recurring events into database rows.
- Study-method recommendation and timers. Those belong to Spec 015.
- Post-study feedback and method adaptation. Those belong to Spec 016.

## Relationship to Existing Specs

- Spec 007 is legacy. Its `available_hours_per_week`, manual `priorities` and `max_session_minutes` inputs are deprecated for the new planning flow.
- Spec 013 established persistent `academic_events`, authenticated calendar CRUD, extraction preview, and the current month/weekly agenda UI. Spec 017 extends `academic_events` instead of replacing it.
- Spec 014 remains authoritative for deterministic priority scoring. Spec 017 consumes ranked priorities and allocates time for them.
- Spec 015 owns actual learner-started study activities and the study-method catalog at `backend/app/knowledge/study_methods/`.
- Spec 016 owns post-activity feedback and method adaptation.

Planned study blocks must not be called completed study sessions. They reserve time only.

## Domain Distinction

### Planned Study Block

A planned study block is a reserved calendar interval generated by weekly planning.

It contains:

- discipline;
- content when available;
- assessment or deadline when relevant;
- reason;
- priority evidence snapshot;
- date;
- start time;
- end time.

It does not define the cognitive study method. It does not imply that the learner actually studied. It may be edited, moved, cancelled or completed as a calendar block state, but completion of a block is not the same thing as completing a study activity.

### Study Activity

A study activity is the actual execution started by the learner. It may use Pomodoro, retrieval practice, self-explanation, worked examples or another allowed method from Spec 015. It records actual duration and post-study feedback from Spec 016.

Planned blocks may provide context for starting a study activity later, but they do not replace activity records.

## Current Implementation Constraints

The current calendar API uses:

- `GET /api/calendar/events`;
- `POST /api/calendar/events`;
- `PATCH /api/calendar/events/{event_id}`;
- status transitions for confirm, complete and cancel;
- `AcademicEventCreate`, `AcademicEventUpdate` and `AcademicEventRead`;
- persistent `academic_events` records isolated by `user_id`.

The current frontend `CalendarPage` renders a month grid and a separate "Agenda semanal" below it, and calls the legacy `POST /api/study-plans/generate` route with fixed sample availability, empty priorities and `max_session_minutes`.

Spec 017 must preserve existing event CRUD while extending the model in the smallest auditable way.

## Weekly Planning Input

Remove from the new UI and new API:

- manual priority selector per discipline;
- required weekly hours field as an independent value;
- maximum duration per session.

Replace them with daily availability windows.

Example:

```json
{
  "week_start": "2026-07-13",
  "availability": [
    { "weekday": "monday", "start_time": "18:00", "end_time": "20:00" },
    { "weekday": "wednesday", "start_time": "14:00", "end_time": "16:00" },
    { "weekday": "saturday", "start_time": "09:00", "end_time": "12:00" }
  ],
  "excluded_discipline_ids": [],
  "objective_text": "Review for next week's exams."
}
```

The backend must calculate:

- available minutes per window;
- total available minutes per day;
- total weekly availability.

The weekly total is derived from windows and must not be typed independently.

The learner may:

- exclude a discipline;
- mark a window as unavailable;
- add multiple windows on the same day;
- remove or edit a window;
- provide an optional weekly objective.

Validation:

- end time must be later than start time;
- overlapping windows on the same day must be rejected or merged deterministically;
- duplicate windows must not be stored;
- timezone remains `America/Sao_Paulo` unless explicitly configured later;
- availability is isolated by authenticated user;
- unavailable windows are excluded from totals.

## Automatic Priority

The backend calculates authoritative priority. The learner must not assign numeric priority manually.

Priority factors include, when evidence exists:

- assessment deadline proximity;
- assessment weight;
- current mastery gap;
- unfinished associated content;
- prerequisite dependencies;
- previous assessment performance;
- academic risk;
- overdue work;
- estimated workload;
- whether enough time remains before the assessment.

The score must be deterministic and auditable. The LLM may explain or summarize the rationale, but may not define the authoritative score, invent evidence, or override backend deadlines.

Every ranked item must expose:

- priority score;
- priority band;
- evidence used;
- missing evidence;
- concise reason;
- relevant assessment or deadline.

Keep these concepts separate:

- priority: what should be studied first;
- difficulty: estimated cognitive challenge;
- method fit: how the content may be studied.

## Planned Block Generation

The planner allocates study blocks inside supplied availability windows.

Rules:

- never schedule outside an availability window;
- never overlap existing opaque calendar events;
- never schedule preparation after the related assessment;
- use the available window as the upper boundary;
- a block may occupy all or part of a window;
- split a window only when needed to allocate different priorities;
- avoid very small fragmented blocks;
- expose unallocated priorities when capacity is insufficient;
- preserve deterministic behavior without an LLM.

The backend must define an internal minimum useful block. Default:

```text
minimum_useful_planned_block_minutes = 30
```

This is a scheduling constraint, not Pomodoro and not a study-method recommendation. It is configurable in the backend but not exposed as "maximum session duration".

Generated blocks are initially a preview. The user must review and confirm before persistence.

## Persistence

Confirmed planned blocks must be persisted as academic calendar events with a distinct origin:

```text
source = "study_plan"
event_type = "study_block"
```

If the existing `CalendarEventType` enum cannot accept `study_block` compatibly, use `activity` temporarily with a normalized `origin = "study_plan"` or `source = "study_plan"` field, then migrate to a first-class type.

Persisted fields:

- `study_plan_id`;
- `discipline_id`;
- `content_id` when available;
- `assessment_id` when relevant;
- priority score snapshot;
- priority band snapshot;
- priority reason snapshot;
- priority evidence snapshot;
- planned start;
- planned end;
- state;
- generated_at;
- algorithm version;
- source fingerprint for idempotency.

Regeneration must not silently duplicate confirmed blocks.

Define explicit behavior:

- regenerating an unconfirmed preview replaces only that preview;
- regenerating after confirmation creates a new preview and reports existing confirmed blocks;
- manually edited blocks are preserved unless the user explicitly cancels or replaces them;
- obsolete generated blocks may be cancelled, not deleted, after confirmation;
- conflicts introduced after generation are detected before confirmation and reported.

## Recurrence Model

Manual events support recurrence.

Supported recurrence frequencies:

- does not repeat;
- daily;
- weekly;
- every two weeks;
- monthly;
- yearly;
- custom weekly recurrence with selected weekdays.

Termination:

- never;
- on selected end date;
- after selected number of occurrences.

Use an RFC 5545 compatible representation where practical, such as RRULE, while exposing a safe product-level schema.

Examples:

```text
RRULE:FREQ=WEEKLY;BYDAY=MO,WE;UNTIL=20261231T235959Z
RRULE:FREQ=WEEKLY;INTERVAL=2;BYDAY=TU
RRULE:FREQ=YEARLY
```

Requirements:

- store the recurrence rule on the event series;
- generate occurrences for calendar range queries;
- avoid permanently creating unlimited rows;
- retain timezone semantics;
- support editing and deletion of the entire series;
- support editing and deletion of a single occurrence;
- document that "this and following" edits may be deferred if too large for the first implementation.

Recurring study-plan blocks are out of scope unless explicitly generated from recurring availability for a bounded week preview. The system must not create endless recurring study blocks.

## API Contracts

### Availability Summary

```http
POST /api/study-plans/availability/summary
```

Request:

```json
{
  "week_start": "2026-07-13",
  "timezone": "America/Sao_Paulo",
  "windows": [
    { "weekday": "monday", "start_time": "18:00", "end_time": "20:00", "available": true }
  ]
}
```

Response:

```json
{
  "timezone": "America/Sao_Paulo",
  "daily_totals": { "monday": 120 },
  "weekly_total_minutes": 120,
  "normalized_windows": []
}
```

### Generate Preview

```http
POST /api/study-plans/weekly-preview
```

Response includes:

- preview id;
- week start;
- availability totals;
- ranked priorities;
- planned blocks;
- unallocated priorities;
- conflicts;
- warnings;
- algorithm version.

### Confirm Preview

```http
POST /api/study-plans/{study_plan_id}/confirm
```

Response includes persisted calendar events and skipped or conflicted blocks.

Confirmation must be idempotent. Repeating the same confirmation request must not create duplicate events.

### Calendar Range Query

```http
GET /api/calendar/events?start_at=2026-07-13T00:00:00-03:00&end_at=2026-07-19T23:59:59-03:00
```

The response must include:

- one-time events;
- expanded recurring occurrences within the requested range;
- confirmed planned study blocks;
- stable occurrence ids for recurring instances;
- source/origin fields that distinguish assessment, manual event and planned study block.

### Create Manual Recurring Event

```http
POST /api/calendar/events
```

Request adds:

```json
{
  "title": "Study group",
  "event_type": "activity",
  "start_at": "2026-07-14T18:00:00-03:00",
  "end_at": "2026-07-14T19:00:00-03:00",
  "recurrence": {
    "frequency": "weekly",
    "interval": 1,
    "weekdays": ["tuesday", "thursday"],
    "ends": { "mode": "on_date", "until": "2026-12-31" }
  }
}
```

Update and delete routes must accept a scope:

- `this_occurrence`;
- `entire_series`;
- `this_and_following`, if implemented.

If `this_and_following` is deferred, the API must return a friendly unsupported-scope response and the UI must not expose the option.

## Persistence Strategy

Prefer the smallest auditable extension to `academic_events`:

- add recurrence fields or a recurrence series table;
- add fields for `origin`, planned block metadata and source fingerprint;
- add indexes by `user_id`, `start_at`, `end_at`, `source`, `study_plan_id` and `recurrence_series_id`;
- store recurrence exceptions without duplicating every future occurrence.

Recommended initial model:

- `academic_events`: base event or single event row;
- `academic_event_recurrence`: one row per series with RRULE/product schema;
- `academic_event_occurrence_exceptions`: cancellations or edited occurrence overrides;
- `study_plan_runs`: preview/confirmation audit;
- `planned_study_blocks`: generated block audit, linked to persisted calendar event after confirmation.

Migrations must:

- keep existing `academic_events`;
- add nullable columns first;
- backfill existing manual events as `origin = "manual"`;
- backfill assessment-derived events as their current source;
- leave existing legacy study plan sessions untouched;
- preserve user isolation.

## Backward Compatibility

Current `StudyPlanSession` behavior is legacy.

Migration strategy:

- existing stored generated sessions are not deleted;
- mark or interpret them as legacy;
- hide them from the new planning form where appropriate;
- optionally display legacy blocks in the calendar with a legacy marker;
- new plans use planned study blocks persisted through calendar events;
- existing APIs remain temporarily available or return a deprecation warning;
- the frontend no longer asks for manual priority or maximum session duration.

No current user data may be destroyed.

## Calendar UX

The calendar page keeps the navigation label "Calendário" and replaces the duplicated layout with a view switch:

- Month;
- Week.

Do not render a separate weekly agenda underneath the monthly calendar.

Month view:

- conventional month grid;
- compact event indicators;
- click a day to inspect its events;
- distinguish assessment, manual event and planned study block.

Week view:

- seven-day time grid or clear chronological day columns;
- display start and end times;
- display planned blocks in their actual time positions;
- allow moving or editing when supported;
- show conflicts clearly;
- include all confirmed events and study blocks.

When a weekly plan is confirmed:

- blocks immediately appear in Month and Week views;
- day details include them;
- refresh preserves them;
- the same block does not appear as both a calendar event and a separate agenda item.

Manual event form:

- adds recurrence controls;
- validates recurrence before submit;
- hides unsupported edit/delete scopes.

Weekly planning form:

- collects availability windows by day;
- shows derived daily and weekly totals;
- allows exclusions and optional objective text;
- does not show manual priority selectors;
- does not show maximum session duration;
- shows preview before confirmation.

## Guardrails

- No planned block outside availability.
- No block overlapping opaque existing events.
- No preparation after the relevant assessment.
- No invented dates, weights, grades, content, mastery or workload.
- No LLM overwrite of priority or allocation.
- No raw technical errors exposed to the frontend.
- No cross-user access to events, availability, plans or recurrence series.
- No infinite event materialization.
- No Pomodoro duration used to generate calendar blocks.
